01 · Core model

One control plane.

Workers produce evidence. The orchestrator alone interprets that evidence through the router and commits a permitted edge in the task state machine.

I

Single canonical state

The root task occupies exactly one lifecycle at a time.

II

Exclusive authority

Only the orchestrator may commit a lifecycle transition.

III

Closed edges

The router chooses from an explicit allowlist. It never invents an edge.

IV

Persistent evidence

Actors are disposable; canonical artifacts and state survive restarts.

02 · Primary flow

The canonical path.

A normal code-changing task moves left to right. Explicit branches handle plan review, no-change work, suspension, cancellation, and targeted rollback.

Code-changing task Scroll horizontally on narrow screens

Plan review branch

Low-confidence canonical plans go from planning to review before implementation or no-change verification.

No-change branch

Advisory or diagnostic work moves from planning directly to verification, then completes without delivery.

Backward routing

Findings return to the earliest lifecycle that owns the invalid assumption or artifact.

03 · Lifecycles

Eleven deliberate states.

Expand a state to see its contract. Router, blocked, rework, failed, and actor execution statuses are intentionally not lifecycle nodes.

01 / REQUIRED

Intake

Normalize the request into an actionable task contract.

Entry

Dependency-ready task assigned, or contract changed.

Output

Objective, acceptance criteria, scope, constraints, authority, and deliverable.

02 / REQUIRED

Context gathering

Ground the task in relevant system evidence.

Entry

Current task contract exists.

Output

Referenced files, behavior, history, instructions, dependencies, and unknowns.

03 / REQUIRED

Analysis

Model impact, regression scope, blast radius, and risk.

Entry

Grounded context package exists.

Output

Impact report, system graph, change-surface matrix, and verification map.

04 / REQUIRED

Planning

Create exactly one canonical execution plan.

Entry

Current impact analysis exists.

Output

Ordered approach, affected areas, criteria mapping, mitigations, and checks.

05 / CONDITIONAL

Implementation

One implementer executes the canonical plan.

Entry

Current plan exists and changes are required.

Output

Change set, tests, deviations, discoveries, and local checks.

06 / PARAMETERIZED

Review

Independently review a plan or implementation.

Entry

Reviewable target and revision exist.

Output

All findings, coverage, impact reconciliation, confidence, and outcome.

07 / REQUIRED

Verification

Prove acceptance and regression expectations.

Entry

Review passed, or no implementation was required.

Output

Checks, results, evidence, exceptions, and completed verification map.

08 / CONDITIONAL

Delivery

Publish the verified revision as a pull request.

Entry

Final change passed local verification.

Output

Branch, commit, PR, description evidence, limitations, and revision match.

09 / SUSPENDED

Awaiting input

Pause for authority, credentials, approval, or information.

Entry

Safe progress requires unavailable input.

Resume

New input triggers router reevaluation; prior work is not blindly resumed.

10 / TERMINAL

Completed

Every applicable gate passed for the current revision.

Entry

Completion invariants all hold.

Exit

None. Terminal states have no outgoing edges.

11 / TERMINAL

Cancelled

An authorized actor explicitly stopped the task.

Entry

Explicit authorized cancellation event.

Exit

None. Failure or uncertainty never implies cancellation.

05 · Router

Rules first. Judgment second.

The router evaluates persisted evidence after an actor result, interruption, new input, cancellation, or restart recovery. It never transitions mid-run.

Hard invariants

Protect safety, authority, freshness, and terminal-state rules.

Allowed edges

Enumerate only declared outgoing transitions.

Guards and policy

Evaluate lifecycle gates, task policy, and retry limits.

Reasoning fallback

Use one high-reasoning evaluation only when rules do not select one edge.

Human input

Enter awaiting input when material ambiguity remains.

Four routing signals

Risklow · medium · high — consequence and blast radius
Confidencelow · medium · high — evidence strength and completeness
Scopelocal · cross-system — structural reach
Attemptsconsecutive failures for the same issue and lifecycle

07 · System view

Review the whole effect.

Analysis predicts system impact. Review reconciles it with the actual change. Verification attaches final evidence. Even trivial tasks produce minimal forms.

Required visual 01

System impact graph

Changed areas, callers, downstream dependencies, data stores, external systems, and boundaries.

Required visual 02

Change-surface matrix

Affected area × impact type × risk × expected modification × regression concern.

Required visual 03

Verification map

Acceptance criteria and identified risks mapped directly to checks and final evidence.

08 · Persistence

Restart from evidence.

The task record holds the canonical lifecycle, artifacts, invocation, attempts, last route, and transition history. Large artifacts are referenced, not embedded.

Persist result

Store the actor envelope before interpreting it.

Evaluate router

Apply invariants, allowed edges, guards, and policy.

Commit transition

Atomically store the route and new lifecycle.

Dispatch actor

Create the next disposable invocation only after commit.

09 · Terminal gate

Completion is proven.

A PR alone is not completion. Every applicable artifact and assurance gate must match the current task revision.

Complete
  • Current task contract and acceptance criteria are satisfied.
  • No required canonical artifact is missing or stale.
  • Applicable independent implementation review passed.
  • Authoritative verification passed for the final revision.
  • When delivery is required, the PR references that exact revision.
  • No unresolved blocker or required approval remains.
